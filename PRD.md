# 🧠 私人笔记网页小程序 PRD（V1.0）

## 一、产品概述

### 1.1 产品名称

私人本地笔记系统（暂定）

### 1.2 产品定位

一款运行在 Windows 本地环境的轻量级网页笔记工具，面向单用户使用，支持文本与代码混合记录，并提供基础的历史版本管理能力。

### 1.3 核心价值

* 快速记录学习、实验和项目过程中的关键信息
* 用统一方式管理文本内容与代码片段
* 保证数据本地可控、结构清晰、后续易扩展

### 1.4 产品原则

本产品 V1 必须始终坚持以下原则：

* **简单**：先完成最小可用闭环，不追求复杂功能
* **可维护**：优先选择清晰稳定的技术方案与数据结构
* **可迭代**：为后续标签、搜索、导出、数据库升级预留空间

### 1.5 目标用户

* 单用户（开发者本人）

### 1.6 核心使用场景

* 学习记录（如爬虫、编程、框架使用）
* 实验/项目笔记
* 代码片段整理与复用

---

## 二、产品目标

### 2.1 核心目标

V1 目标聚焦于一个完整且稳定的个人笔记闭环：

* 快速创建并记录笔记
* 支持文本块与代码块混合编辑
* 数据本地持久化存储
* 支持历史版本查看

### 2.2 非目标（V1 不做）

以下能力不纳入 V1 范围：

* 用户登录系统
* 多端同步
* 云存储
* 权限管理
* 在线协作
* 复杂富文本排版系统

### 2.3 V1 成功标准

如果 V1 能稳定完成以下事情，即视为达成目标：

* 用户可以创建、编辑并保存笔记
* 用户可以在同一篇笔记中混合使用文本块和代码块
* 关闭页面或重启程序后，笔记数据仍可恢复
* 每次保存后都可以看到可追溯的历史版本

---

## 三、功能范围

## 3.1 笔记管理模块

### 3.1.1 创建笔记

* 用户可以新建笔记
* 默认命名规则：`未命名笔记-时间戳`
* 新建后立即进入编辑状态

### 3.1.2 查看笔记列表

* 左侧列表展示所有笔记
* 每项显示：
  * 标题
  * 最近更新时间
* 默认按更新时间倒序排列
* 点击某条笔记后，在右侧加载对应内容

### 3.1.3 编辑笔记

* 每篇笔记由多个块（Block）组成
* V1 支持两种块类型：
  * 文本块（Markdown）
  * 代码块
* 用户可以新增块、删除块
* 块的展示顺序即保存顺序
* 笔记标题支持直接编辑

### 3.1.4 保存笔记

* V1 必须提供**手动保存按钮**
* 自动保存作为后续增强能力，V1 可不实现
* 保存后更新：
  * 当前笔记内容
  * `updated_at` 时间
  * 历史版本快照

---

## 3.2 历史版本模块

### 3.2.1 自动生成版本

* 每次手动保存时，自动生成一个历史版本
* 版本内容至少包含：
  * 保存时间
  * 当时的块内容快照

### 3.2.2 查看历史版本

* 用户可以打开历史记录面板
* 按时间倒序展示历史版本列表
* 点击某个版本后，可查看当时的笔记内容

### 3.2.3 恢复版本

* V1 中该能力为**可选项**
* 若实现，恢复操作应将历史版本内容复制为当前版本，再由用户手动保存确认

---

## 3.3 编辑器模块（核心）

### 3.3.1 文本块

* 支持基础 Markdown 语法输入
* V1 以编辑体验为主
* 实时预览可简化为：
  * 暂不实现，或
  * 使用轻量预览切换

### 3.3.2 代码块

* 支持代码输入
* 支持语法高亮
* 支持语言选择
* V1 默认支持：
  * Python
  * JavaScript

### 3.3.3 编辑体验要求

* 编辑交互应尽量轻量，不引入复杂块嵌套
* 块的新增与切换逻辑要清晰直观
* 优先保证稳定输入、稳定保存，而不是追求富交互

---

## 四、非功能需求

### 4.1 性能

* 本地加载时间应控制在 1 秒内（个人使用规模下）
* 常见操作无明显卡顿
* 保存操作应及时反馈结果
* 预览渲染应避免在输入过程中无节制重复刷新

### 4.2 数据安全

* 数据仅存储在本地
* 不上传任何服务器
* 数据结构应尽量可读，便于后续人工备份或迁移

### 4.3 可用性

* 界面简洁直观
* 新建、编辑、保存、查看历史等核心操作路径清晰
* 尽量减少用户理解成本

### 4.4 可维护性

* 代码模块职责清晰
* 前后端边界明确
* 数据结构保持简单，不为未来功能过度设计

---

## 五、技术方案

### 5.1 技术选型

| 模块 | 技术方案 |
| --- | --- |
| 前端 | HTML + CSS + JavaScript |
| 编辑器 | CodeMirror（推荐） |
| 后端 | Flask 本地服务 |
| 存储 | 本地 JSON 文件 |

### 5.2 技术方案说明

为保证“简单、可维护、可迭代”，V1 推荐采用**轻量前后端分离**：

* 前端负责页面展示与交互
* Flask 仅负责本地文件读写与数据接口
* 数据存储先使用 JSON 文件，不引入数据库

这样做的好处：

* 浏览器层无需直接处理复杂本地文件权限问题
* 后续如需切换到 SQLite，也可以在后端层平滑演进
* 整体结构足够简单，适合个人项目持续迭代

### 5.3 系统架构

```text
浏览器页面
   ↓
Flask 本地服务
   ↓
本地 JSON 文件
```

### 6.1 主界面布局

```text
---------------------------------
| 左侧笔记列表 | 右侧编辑区域 |
---------------------------------
```

### 6.2 页面组成

#### 左侧区域

* 笔记列表
* 新建按钮
* 当前选中笔记高亮

#### 右侧区域

* 笔记标题输入框
* Block 编辑区
* 保存按钮
* 历史记录按钮

### 6.3 交互优先级

V1 优先保证以下交互顺畅：

1. 新建笔记
2. 添加文本块/代码块
3. 编辑内容
4. 手动保存
5. 查看历史版本

不在 V1 中引入复杂动画、拖拽编排或高度自定义布局。

## 七、数据存储设计

### 7.1 存储方式

V1 使用本地 JSON 文件保存笔记内容。

### 7.2 文件结构建议

```text
/data/
   note_001.json
   note_002.json
```

### 7.3 单篇笔记数据结构示例

```json
{
  "id": "note_001",
  "title": "学习记录",
  "blocks": [
    { "id": "block_001", "type": "text", "content": "今天学习了爬虫" },
    { "id": "block_002", "type": "code", "language": "python", "content": "print('hello')" }
  ],
  "versions": [
    {
      "time": "2026-04-16 10:00:00",
      "blocks": [
        { "id": "block_001", "type": "text", "content": "今天学习了爬虫" },
        { "id": "block_002", "type": "code", "language": "python", "content": "print('hello')" }
      ]
    }
  ],
  "created_at": "2026-04-16 09:30:00",
  "updated_at": "2026-04-16 10:00:00"
}
```

### 7.4 数据设计原则

* 当前内容与历史版本保存在同一文件中，方便单文件管理
* 字段命名保持统一、直接
* 历史版本持久化优先保留 `document` 与必要元数据，运行时再按需还原 `blocks`
* 通过原子写入降低写入过程中损坏文件的风险

* 可以创建新笔记

* 可以新增文本块与代码块

* 保存后可在本地 JSON 文件中看到完整数据
* 重启本地服务后，历史笔记仍可正常加载
* 每次保存都能生成一条可查看的历史版本记录

* 核心操作路径清晰，无明显理解障碍
* 常规使用过程中无明显卡顿或频繁报错

## 九、开发里程碑

* 请求期间关键按钮禁用，避免重复操作

## 十、风险与应对

| 风险 | 说明 | 应对方案 |
| --- | --- | --- |
| 本地文件读写处理复杂 | 浏览器直接操作本地文件存在限制 | 通过 Flask 提供本地文件 API |
| 历史版本导致文件膨胀 | 单文件存储版本可能逐渐变大 | V1 先接受，后续按需拆分版本存储 |
| 编辑器集成复杂度偏高 | 富编辑器能力容易超出 V1 范围 | 优先接入成熟组件，只保留核心能力 |
| 需求扩张过快 | 容易提前加入同步、搜索、标签等功能 | 坚持最小可用闭环，按阶段推进 |

# ✅ 总结

本产品 V1 的本质不是做一个“大而全”的笔记系统，而是先做一个：

**能用、稳定、结构清晰、方便后续扩展的个人本地笔记工具。**

因此，V1 的优先级始终是：

1. 先把创建、编辑、保存、历史查看这条主链路做通
2. 保持架构简单，避免过早复杂化
3. 为未来迭代留出空间，但不提前实现未来需求
